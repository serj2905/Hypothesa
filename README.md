# Бизнес-анализ (Business Understanding) — проект «Мичер»

> Домашнее задание №1. Первый этап процесса разработки сервисов машинного обучения для — интеллектуального бота-интервьюера для адаптивных социологических опросов на базе большой LLM.
>
---
## 1. Цели и скоуп проекта

### Организационная структура: кто участвует и кто пользователь

- **Заказчик сервиса** — организация, которой нужна непрерывная адаптивная обратная связь: исследовательские/социологические центры, медицинские учреждения, бизнес (банки, ритейл, сервисные компании).
- **Основные пользователи** — две группы. Во-первых, **респонденты**, проходящие опрос. Во-вторых, **аналитик/исследователь заказчика**, который запускает исследование, настраивает гиперпараметры и получает на выходе сводку с рейтингом категорий.

### Бизнес-цель проекта

Превратить опрос из статичной анкеты в **самообучающийся цикл**: система должна автоматически порождать гипотезы о том, что волнует респондентов, и тут же проверять их в следующих интервью — без ручного перестроения анкеты аналитиком. Для бизнеса это означает получение актуальной, релевантной обратной связи с минимальной длиной анкеты и без ручной разметки тысяч открытых ответов.

### Существующие решения и чем они не устраивают

Уже существуют классические форматы сбора данных — бумажные и цифровые анкеты, а также живые интервью. Их принципиальные ограничения:

- **Статический сценарий.** Список вопросов фиксируется до начала исследования и не меняется по мере поступления данных, поэтому исследование не успевает за реальной повесткой респондентов.
- **Невозможность обработки открытых ответов в масштабе.** Когда ответов тысячи, человек не в состоянии вручную выявить паттерны и актуализировать вопросник.
- **«Раздутые» опросники.** Стремление охватить всё сразу даёт длинные формы, которые вызывают негатив у респондентов и снижают качество ответов.

## 1.1. Текущая ситуация (Assessing current solution)

### Достаточно ли ресурсов для проекта

**Железо / вычисления.** Базовый сценарий использует open-source LLM (например, семейства Llama/Qwen).

**Данные и доступ к ним.** Размеченного датасета на старте нет — данные накапливаются в ходе самих интервью. Состояние диалогов хранится локально, итоговые результаты в csv формате.

**Эксперты заказчика.** Для постановки вопросов и интерпретации категорий желательно привлечение предметного эксперта заказчика, но базовый цикл работает автономно.

**Технологический стек:** LLM (llama/qwen), **BERTopic** для тематической кластеризации открытых ответов, `joblib` для состояния, `pandas` и `collections.defaultdict` для анализа, `plotly` для визуализации.

### Вероятные риски и план их уменьшения

| Риск | Влияние | План снижения |
|---|---|---|
| **Не уложиться в сроки** | Срыв поставки сервиса заказчику | Итеративная разработка по фазам CRISP-DM; демоинтеграция в Telegram собрана за ~2 часа, что подтверждает быструю поставку MVP |
| **Малое количество / плохое качество данных** | Невозможность построить осмысленные категории и рейтинг | Накопление данных в самом цикле опроса; уточняющие вопросы повышают информативность открытых ответов; суммаризация чистит шум |
| **Закономерности отсутствуют / результаты не интересны заказчику** | Бесполезный для бизнеса вывод | Рейтингование отсеивает неактуальные темы (рейтинг ≤ 0 → выбывает); в сценарий попадают только реально волнующие респондентов категории |
| **Артефакты генерации LLM** (модель отвечает за пользователя, нарушает формат нумерации, не завершает диалог) | Срыв парсинга и постобработки | Шаблонные инструкции в промпте, structured output / function calling, ручные команды-подсказки, перезапуск диалога командой `/start` |

## 1.2. Решаемые задачи с точки зрения аналитики (Data Mining goals)

Бизнес-цель переводится в постановку в технических терминах. Работа Мичера — замкнутый цикл из пяти ML-подзадач:

1. **Проведение интервью.** По базовому промпту бот задаёт закрытые (возраст, город) и открытые вопросы; на каждый открытый LLM генерирует `promt_open_q` уточняющих вопросов на основе ответов конкретного респондента.
2. **Суммаризация.** Ответы делятся по вопросам; для каждого открытого ответа LLM извлекает суть в виде структурированного списка проблем или положительных аспектов.
3. **Формирование категорий (тематическая кластеризация).** Суммаризированные ответы объединяются и группируются в тематические категории. Здесь предполагается использование **BERTopic**: ответы кодируются sentence-эмбеддингами, понижается размерность (UMAP), кластеризуются (HDBSCAN), а тема каждого кластера описывается через c-TF-IDF (опционально — с генерацией читаемого названия темы через LLM). Это даёт воспроизводимую и масштабируемую кластеризацию вместо целиком промптовой группировки. Каждой новой категории присваивается стартовый рейтинг `raiting_param`.
4. **Рейтингование.** Для каждого нового ответа проверяется упоминание категории: упоминается → +1, не упоминается → −1; при рейтинге ≤ 0 категория исключается как неактуальная. Принадлежность ответа к теме определяется по близости его эмбеддинга к центроиду кластера BERTopic либо через `transform()` на новых документах.
5. **Динамическое переформирование промпта.** По принципу «колоды карт» (категория представлена в колоде столько раз, каков её рейтинг) случайно вытягиваются `promt_open_q_reit` категорий и включаются в следующий сценарий — чем выше рейтинг, тем чаще тема попадает в опрос.

> **Почему BERTopic.** Группировка категорий «в лоб» промптом нестабильна: на разных прогонах LLM выдаёт разные наборы и формулировки категорий, что мешает считать рейтинг между сессиями. BERTopic даёт детерминированную (при фиксированном seed) кластеризацию на эмбеддингах, умеет инкрементально присваивать новые документы к существующим темам (`transform`) и сливать близкие темы (`reduce_topics`/`merge_topics`) — это напрямую ложится на логику накопления и актуализации рейтинга категорий. LLM при этом остаётся для интервью, суммаризации и читаемого именования тем.

### Метрика оценки результата

Задача относится к классу **обработки естественного языка без объективного эталона (ground truth)** — заранее «правильного» набора категорий не существует, он порождается самой моделью. Поэтому классические метрики (Accuracy, AUC, F-мера, RMSE) напрямую неприменимы. Качество оценивается комбинированно:

- **Динамика рейтинга категорий** как основной измеримый показатель: насколько частота упоминания темы респондентами соответствует росту её рейтинга. Визуализируется столбчатыми диаграммами `data/analim{N}.png`.
- **Качество кластеризации BERTopic** — здесь применимы полуобъективные метрики: **topic coherence** (например, $c_v$), доля «шумовых» документов (кластер `-1` у HDBSCAN), а также внутрикластерная компактность / межкластерная разделимость (silhouette по эмбеддингам).
- **Релевантность и неизбыточность категорий** — экспертная (субъективная) оценка: категории осмысленны, не дублируются, отражают содержание ответов и относятся к предмету исследования.
- **Длина анкеты для респондента** — контролируется, чтобы охват тем рос без раздувания формы.

### Критерий успешности модели

Объективного числового порога (вроде «AUC ≥ 0.75») в данной задаче нет, поэтому критерий формулируется так:

- система **самостоятельно** формирует и ранжирует категории из открытых ответов без ручной разметки;
- актуальные темы устойчиво чаще попадают в последующие интервью, неактуальные — выбывают;
- сводка на выходе (`results.csv`, рейтинг категорий, диаграммы) пригодна для принятия управленческих решений заказчиком;
- гиперпараметры можно менять, не останавливая исследование.

### Управляемые гиперпараметры

| Параметр | Назначение |
|---|---|
| `prompt` | Базовый шаблон сценария интервью |
| `open_quastions` | Список номеров открытых вопросов |
| `summing_promt_dict` | Промпты для суммаризации ответов по каждому вопросу |
| `raiting_param` | Стартовый рейтинг новой категории |
| `promt_open_q` | Количество уточняющих вопросов на основе ответов респондента |
| `promt_open_q_reit` | Количество «подтасованных» вопросов из рейтингованных категорий |

## 2. Идеи для развития проекта

Ниже — направления, которые можно интегрировать в Мичер. Сгруппированы по приоритету и эффекту.

### Ядро аналитики

- **BERTopic как основной слой кластеризации** (вместо промптовой группировки). Даёт стабильные, воспроизводимые темы, инкрементальное присвоение новых ответов (`transform`), слияние близких тем и иерархию тем — ровно то, что нужно для устойчивого рейтинга между сессиями.
- **Семантический рейтинг вместо бинарного.** Сейчас упоминание темы даёт ±1. Можно перейти к взвешиванию по косинусной близости эмбеддинга ответа к теме и по силе тональности — тогда сильно выраженная проблема поднимает тему быстрее слабого упоминания.
- **Анализ тональности (sentiment) на уровне категории.** Отделять «болит сильно» от «упомянул вскользь»; строить не только частоту, но и эмоциональный вес темы.
- **Динамические темы во времени (Dynamic Topic Modeling).** BERTopic умеет строить эволюцию тем по времени — можно показывать заказчику, как повестка меняется по неделям/месяцам, а не только текущий срез.

### Качество интервью и данных

- **Structured output / function calling** для LLM вместо парсинга по подстроке «свидания». Убирает целый класс артефактов: модель возвращает строгий JSON (номер вопроса, тип, флаг завершения), что чинит хрупкую постобработку.
- **Адаптивная длина интервью.** Останавливать опрос, когда новые ответы перестают приносить новые темы (information saturation / насыщение) — экономит время респондента.
- **Детекция некачественных ответов** (троллинг, односложные «нормально», бот-ответы) с переспросом или мягким исключением из аналитики.
- **Защита от деградации «колоды».** Подстраховать взвешенную выборку от нулевых/отрицательных весов и добавить ε-исследование (epsilon-greedy): иногда задавать вопрос по «холодной» теме, чтобы не закапываться только в топовые категории.

### Продукт и охват

- **Дашборд для заказчика** (живая страница с рейтингом тем, динамикой, фильтрами по городу/возрасту) вместо разовых PNG и словаря в чат.
- **Мультиканальность.** Помимо Telegram — WhatsApp/VK/веб-виджет; в перспективе **голосовой интервьюер** (STT на входе, TTS на выходе), как вектор роста проекта.
- **Сегментация респондентов.** Считать рейтинг тем отдельно по сегментам (возраст, город, частота пользования услугой) — заказчик видит, что разные группы волнует разное.
- **A/B-тестирование промптов и гиперпараметров** прямо в проде: сравнивать формулировки вопросов по доле завершённых интервью и информативности ответов.

### Инженерия и надёжность

- **Перенос состояния с `joblib`-файлов в БД** (PostgreSQL / SQLite + слой ORM). Снимает гонки при параллельных диалогах и упрощает аналитику.
- **Кэширование эмбеддингов и асинхронные вызовы LLM** — главное узкое место сейчас (цикл до 10–20 минут). Батчинг запросов и фоновые задачи (Celery/RQ) сократят время в разы.
- **Вынос секретов в `.env` и ротация ключей** — устранить хранение API-ключа и токена бота в коде.
- **Версионирование данных (DVC)** и логирование экспериментов — как и рекомендует методология на этапах Data Understanding/Preparation.
