# Спринт 1 — Data Layer: надёжная генерация данных

> Цель спринта: убрать классы ошибок в начале пайплайна (парсинг + суммаризация)
> и поставить измеритель качества. EDA по hw_4 проводится **после** этого спринта,
> уже на чистых данных. BERTopic, рейтинг и дашборд — следующими спринтами.

## Контекст железа

- GPU: **RTX 5080, 16 GB VRAM** (Blackwell, sm_120 — нужны свежие сборки Ollama / vLLM с CUDA 12.8+).
- Бюджет VRAM: ~13 GB на веса + ~2-3 GB на KV-кэш и контекст.

## Решение по LLM: генератор и судьи — РАЗНЫЕ семейства

Ключевое архитектурное решение спринта. Причина — **self-preference bias**: если суммаризатор
и Judge 2 (faithfulness) — одна модель, судья завышает оценку собственным галлюцинациям,
и KPI Faithfulness ≥ 0.70 оказывается недостоверным.

| Роль | Модель | Квант | Когда в VRAM |
|---|---|---|---|
| Интервьюер + суммаризатор | Qwen2.5-14B-Instruct | Q4_K_M / Q5_K_M | live (во время опроса) |
| Dev / отладка eval-harness | Qwen2.5-7B-Instruct | Q8 | по необходимости |
| Судьи 1-4 | **Llama-3.1-8B** или **Mistral-Small** (другое семейство!) | Q4/Q5 | batch (после выгрузки генератора) |
| Эмбеддинги (для BERTopic, позже) | paraphrase-multilingual-MiniLM-L12-v2 | — | CPU |
| Тёмная лошадка для судей | Qwen3-30B-A3B (MoE) | Q4 + офлоад | batch, если 8B слабоват |

### Почему это влезает в 16 GB

Генератор и судьи **никогда не загружены одновременно** — судейство это
оффлайн-батч по накопленному `results.csv`, а не live-цикл диалога:

```
Фаза 1 (live):  Telegram-опрос идёт на Qwen-14B        → в VRAM Qwen
Фаза 2 (batch): Qwen выгружен, грузим Llama/Mistral    → судим накопленные диалоги
```

Латентность судей некритична → на фазу 2 при необходимости можно грузить модель
крупнее (30B-A3B / 32B Q4 с частичным CPU-офлоадом). Сильный судья > быстрый судья.

Усиление при наличии ресурсов: **панель судей** (2-3 разные модели голосуют) —
снижает bias и даёт естественную меру согласованности (Judge 4).

## Структурированный вывод

- Ollama: JSON-схема через параметр `format`.
- vLLM: guided decoding (`outlines` / `lm-format-enforcer`) — гарантия валидного JSON
  на уровне декодера. На Blackwell позже — FP8/FP4 эффективнее GGUF-Q4.

## Definition of Done

- Парсинг диалога не зависит от текстовых маркеров; перемешивание полей в `results.csv` исключено.
- Суммаризация всегда возвращает валидный JSON по схеме (format-valid rate = 100%).
- Judge 2 (другое семейство) считает Faithfulness на golden set; baseline зафиксирован.
- Секреты вне репозитория, старые токены отозваны.
