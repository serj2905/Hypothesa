# Домашнее задание 2. Прототип продукта

Проект: **Мичер** - интеллектуальный бот-интервьюер для адаптивных социологических и клиентских опросов на базе LLM.

Цель прототипа - за минимальное время проверить, есть ли у продукта понятная платная ценность для конкретного сегмента, и можно ли технически собрать стабильный цикл: интервью -> суммаризация -> темы -> рейтинг -> обновленный сценарий интервью.

---

## 1. Ключевая гипотеза

**Формат гипотезы**

Если мы предлагаем **адаптивного LLM-бота-интервьюера с автоматическим выделением и рейтингом тем из открытых ответов**
для сегмента **исследовательских и CX-команд в компаниях с регулярными опросами клиентов или граждан**
в сценарии **быстрого исследования удовлетворенности после взаимодействия с сервисом**, 
то мы получим **минимум 2 подтвержденных платных пилота** по модели **usage-based: пакет 500 валидных интервью за 25 000 рублей**
в течение **4 недель**,
потому что у этого сегмента уже есть боль: статичные анкеты быстро устаревают, открытые ответы дорого кодировать вручную, а бизнесу нужен не массив сырых ответов, а понятный рейтинг актуальных тем.

**Что именно проверяем**

- Один сегмент: исследовательские и CX-команды, которые проводят повторяющиеся опросы и анализируют открытые ответы.
- Один сценарий: короткий адаптивный опрос после клиентского или пользовательского опыта.
- Один способ монетизации: оплата за пакет валидных завершенных интервью.
- Главный бизнес-сигнал: готовность взять платный пилот, а не просто похвалить демо.

## 2. Ключевые метрики прототипа

### 2.1. Продуктовые и бизнес-метрики

| Метрика | Baseline | Целевое значение | Как измеряем | Частота |
|---|---:|---:|---|---|
| Paid pilot conversion | 0 платных пилотов | >= 2 платных пилота из 8-10 целевых демо | Фиксируем согласие на пилот, счет, оплату или письменное подтверждение бюджета | В конце 4-недельного теста |
| Completion Rate интервью | 60-70% для длинных статичных форм или первое контрольное измерение | >= 80% валидно завершенных диалогов | Доля диалогов с корректным завершением и сохраненными ответами | Ежедневно и по итогам каждой пачки интервью |
| Analyst time-to-insight | 2-4 часа ручного просмотра 50 открытых ответов | <= 30 минут до топ-5 тем и краткой сводки | Замер времени от окончания сбора ответов до готового отчета | После каждой пачки из 50 интервью |
| Доля полезных тем в отчете | 0 - до первого запуска | >= 70% тем аналитик помечает как полезные для решения | Быстрая оценка аналитиком заказчика: useful / not useful | После каждого отчета |
| Длина интервью | 10-12 минут у расширенной статичной анкеты | <= 8 минут при сохранении обязательных вопросов | Время от первого сообщения до завершения | По каждому диалогу, агрегируем ежедневно |
| Повторный запуск исследования | 0 | >= 1 заказчик хочет повторить запуск на новой группе | Факт повторного сценария или запрос на следующий срез | В конце пилота |

### 2.2. Технические метрики

| Метрика | Baseline | Целевое значение | Как измеряем | Частота |
|---|---:|---:|---|---|
| Question Coverage | 0 до первого запуска | 100% обязательных закрытых вопросов заданы | Парсинг транскрипта по структуре сценария | Каждый диалог |
| Interview Quality Judge | Первый прогон прототипа | >= 0.75 | LLM Judge по adherence, relevance, naturalness, completion | Каждые 20 диалогов |
| Summarization Faithfulness Judge | Первый прогон прототипа | >= 0.70, Faithfulness не равен 0 | LLM Judge сравнивает исходный ответ и суммаризацию | Каждые 20 открытых ответов |
| Topic Coherence C_v | Prompt-only группировка без стабильного baseline | >= 0.50 | `gensim` CoherenceModel для тем BERTopic | После каждой кластеризации |
| Silhouette Score | Первый прогон BERTopic | >= 0.25 | `sklearn.metrics.silhouette_score` по sentence-эмбеддингам | После каждой кластеризации |
| Noise Ratio | Первый прогон BERTopic | <= 20% документов в теме -1 | Доля outlier-документов HDBSCAN | После каждой кластеризации |
| Coverage Ratio топ-5 тем | Первый прогон BERTopic | >= 60% ответов покрыты топ-5 темами | Подсчет присвоенных тем через BERTopic `transform()` | После каждой кластеризации |
| Rank Correlation | Первый прогон рейтинга | Spearman rho >= 0.60 | Корреляция рейтинга тем с частотой упоминаний в сырых ответах | После каждой пачки из 50 интервью |
| Себестоимость валидного интервью | Первый запуск на LLM API | <= 20 рублей за валидное интервью | Логи вызовов LLM + стоимость инфраструктуры / число валидных интервью | Еженедельно |
| Время обработки пачки | Ручная обработка 2-4 часа | <= 15 минут на 50 интервью | Время суммаризации, эмбеддингов, кластеризации и отчета | После каждой пачки |

## 3. Что входит в прототип, а что намеренно не делаем

### Входит в прототип

| Блок | Что делаем | Почему это нужно для проверки гипотезы |
|---|---|---|
| Интерфейс респондента | Telegram-бот или простой web-chat с одним сценарием опроса | Нужно проверить завершение интервью и качество открытых ответов на живых пользователях |
| Сценарий интервью | 3-5 обязательных вопросов, 1-2 открытых вопроса и ограниченное число уточнений | Дает достаточно данных, но не превращает прототип в длинную анкету |
| Суммаризация ответов | LLM выделяет проблемы, плюсы и ключевые тезисы из открытых ответов | Без этого аналитик не получает быстрый time-to-insight |
| Тематическая кластеризация | BERTopic на sentence-эмбеддингах для устойчивых тем | Проверяем, можно ли получить воспроизводимые категории вместо нестабильной prompt-only группировки |
| Рейтинг категорий | Простая логика +1 / -1 и выбор тем по принципу колоды | Это ключевая продуктовая механика адаптивного опроса |
| Базовая аналитика | CSV/SQLite с диалогами, `results.csv`, топ-5 тем, график рейтинга, сводка | Заказчик должен увидеть не технический лог, а управленческий результат |
| Настройка гиперпараметров | `prompt`, список открытых вопросов, число уточнений, стартовый рейтинг | Нужно проверить, может ли аналитик управлять исследованием без переписывания кода |
| Логи и валидация | Сохранение транскриптов, ошибок парсинга, judge-оценок и стоимости | Без логов нельзя понять, почему прототип работает или ломается |

### Не входит в прототип

| Что не делаем | Почему откладываем |
|---|---|
| Полноценный enterprise-кабинет, роли, SSO и сложные права доступа | Это не влияет на проверку платной ценности и займет много времени |
| Поддержку всех каналов: WhatsApp, VK, web-widget, email | Для проверки достаточно одного канала с живыми респондентами |
| Идеальную ML-модель и fine-tuning | На этапе прототипа важнее качество решения бизнес-сценария, а не максимальная точность модели |
| Автоматическую оплату и биллинг | Монетизацию проверяем через платный пилот и ручной счет |
| Высоконагруженную инфраструктуру | Пилот рассчитан на десятки или сотни интервью, не на массовый production |
| Полную автоматизацию аналитического цикла | Часть операций можно пройти вручную как Wizard of Oz, если это быстрее проверяет гипотезу |
| Полную документацию и 100% unit-тестов | Достаточно описания запуска, структуры данных и тестов критических сценариев |
| Долгосрочный retention на 6 месяцев | Для первого прототипа важнее готовность к платному пилоту и повторному запуску |

## 4. Готовые блоки и что делаем сами

### Берем готовое

| Готовый блок | Инструмент | Зачем используем |
|---|---|---|
| Канал интервью | Telegram Bot API / `aiogram` или простой Streamlit-chat | Быстро запускаем живой сценарий без разработки сложного фронтенда |
| LLM для интервью и суммаризации | OpenAI API, Qwen или Llama | Не обучаем модель с нуля, проверяем продуктовую логику |
| Эмбеддинги | `sentence-transformers` | Нужны для кластеризации и поиска близости ответов к темам |
| Тематическая кластеризация | BERTopic + UMAP + HDBSCAN | Получаем воспроизводимые темы и возможность `transform()` для новых ответов |
| Табличная обработка | `pandas`, CSV/SQLite | Быстро храним диалоги, ответы, темы и метрики |
| Визуализация | Plotly / Streamlit | Быстрый дашборд без отдельной frontend-разработки |
| Оценка качества | LLM as Judge с фиксированными rubric-промптами | Позволяет оценивать смысловое качество без ручной разметки всего датасета |
| Хостинг пилота | Yandex Cloud, Render, Railway или локальный сервер для пилота | Не строим production-инфраструктуру до проверки спроса |

### Делаем сами

| Собственный блок | Почему критично делать самим |
|---|---|
| Адаптивная бизнес-логика интервью | Это ядро продукта: какие ответы ведут к каким уточнениям и как сценарий меняется между интервью |
| Механизм рейтинга категорий | Именно он превращает обычный опрос в самообновляющийся цикл гипотез |
| Схема данных диалогов и результатов | От нее зависит, можно ли потом считать метрики, воспроизводить версии и объяснять выводы заказчику |
| Prompt-шаблоны для интервью, суммаризации и judge-оценки | Качество промптов напрямую влияет на завершение диалога, faithfulness и доверие аналитика |
| Дашборд с топ-темами и управленческой сводкой | Это главный артефакт ценности для заказчика, а не просто техническая визуализация |
| Правила остановки и невалидности диалога | Без них нельзя отделить полезные ответы от сломанных, пустых или некачественных данных |

## 5. Главный риск, который проверяет прототип

Главный риск - **не в том, сможет ли LLM провести диалог**, а в том, даст ли адаптивный опрос более ценное и оплачиваемое решение, чем обычная анкета плюс ручной анализ.

Прототип должен проверить три связанные вещи:

1. Респонденты действительно проходят короткое адаптивное интервью до конца.
2. Аналитик доверяет автоматически выделенным темам и рейтингу достаточно, чтобы использовать их в решении.
3. Заказчик готов платить за валидные интервью или пилот, потому что экономит время анализа и быстрее видит актуальную повестку.

Если технически все работает, но аналитик говорит, что топ-темы не помогают принимать решения, продуктовая гипотеза не подтверждена.

---

## 6. Конечный формат прототипа

Финальный вид прототипа для первого теста:

- **Респондентская часть:** Telegram-бот или web-chat, который проводит один адаптивный сценарий интервью.
- **Аналитическая часть:** Streamlit-дашборд с количеством интервью, completion rate, топ-5 темами, рейтингом категорий, графиком динамики и списком примеров ответов по каждой теме.
- **Данные:** `results.csv` или SQLite-таблица с транскриптами, суммаризациями, темами, рейтингами и judge-оценками.
- **Отчет для заказчика:** короткая сводка: какие темы лидируют, какие выбывают, какие сегменты отличаются, что рекомендуется проверить следующим опросом.
- **Настройки:** файл конфигурации с промптом, открытыми вопросами, числом уточнений, стартовым рейтингом и параметрами BERTopic.

## 7. Критерии остановки теста прототипа

### Тест считаем успешным и переходим к следующей итерации, если за 4 недели:

- проведено минимум 50 валидных интервью в одном выбранном сценарии;
- есть минимум 2 подтвержденных платных пилота или эквивалентное письменное подтверждение бюджета;
- Completion Rate >= 80%;
- аналитик заказчика помечает >= 70% топ-тем как полезные;
- Category Quality Judge >= 0.70;
- Summarization Faithfulness Judge >= 0.70;
- Coverage Ratio топ-5 тем >= 60%;
- себестоимость валидного интервью <= 20 рублей;
- заказчик понимает, какое решение он принял бы на основе отчета.

### Гипотезу останавливаем или сильно пересобираем, если:

- после 8-10 целевых демо нет ни одного готового платного пилота;
- Completion Rate стабильно ниже 60% после двух исправлений сценария;
- автоматические темы остаются непонятными аналитику: полезность < 50% или Category Quality Judge < 0.60;
- суммаризация регулярно добавляет факты, которых не было в ответах респондентов;
- стоимость валидного интервью выше 40 рублей и не снижается после оптимизации;
- адаптивность не дает новых тем по сравнению со статичной анкетой;
- заказчик использует отчет только как иллюстрацию, но не видит управленческого решения.

Итоговое решение после теста: **масштабировать**, если есть платная готовность и метрики качества; **пивотить сценарий или сегмент**, если техника работает, но ценность не покупают; **остановить**, если не подтверждены ни ценность, ни качество данных.
